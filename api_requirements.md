# API Requirements — UST × Premera Engagement Portal

This document is the **build-from-scratch requirements spec** for a backend API to replace
the current client-only storage (`localStorage`) in this static React app. It is more
detailed/current than `API.md` (which has drifted from the code — e.g. it still shows a
single `program` block; the app now splits Program Level into **ITO** and **BPO** tabs and
issues carry `riskActions` + `needsPremeraAttention`). Treat **this file** as the
ground truth going forward and update/retire `API.md` once the backend is built.

This was generated by reading the actual current source:
`src/data/portalData.ts`, `src/data/status.ts`, `src/context/PortalDataContext.tsx`,
`src/pages/{ProgramLevel,TowerLevel,Risks,Feedback,Admin}.tsx`,
`src/components/{QuickFeedback,RiskActionsModal,IssuesTable,MilestoneTable}.tsx`.

---

## 1. Why an API is needed — current storage points

The app today persists everything client-side, in three independent `localStorage` keys.
Each one needs a backend equivalent:

| # | Current storage key | Written by | Read by | Needs backend? |
|---|---|---|---|---|
| 1 | `ustpremera_portaldata_v2` | Admin page `save()` | All slide pages via `usePortalData()` | **Yes — primary dataset** |
| 2 | `ustpremera_quickfeedback_v1` | `QuickFeedback` widget (thumbs up/down + comment, per table row) | `QuickFeedback` widget only (own user's browser) | **Yes — needs to be shared** |
| 3 | none (in-memory only, `useState` in `Feedback.tsx`) | Feedback form `submit()` | Feedback page log panel | **Yes — currently lost on refresh, must be persisted** |

There is no auth today (anyone can open `/admin` and overwrite the dataset). The API must
add authentication/authorization that the frontend currently has no concept of.

## 2. Conventions

| Item | Value |
|---|---|
| Base URL | `https://api.<env>.premera-ust.example/v1` |
| Protocol | HTTPS only |
| Format | JSON, `Content-Type: application/json`, UTF-8 |
| Dates (report fields) | Free-text strings exactly as typed in Admin (e.g. `"4/1/2025"`, `"DD/MM/YYYY"`). **Do not format-validate** — they render verbatim on slides. |
| Dates (audit/system fields) | ISO-8601 UTC, e.g. `"2026-06-16T14:32:00Z"` |
| IDs | Opaque stable strings. Client generates `crypto.randomUUID()` client-side (see `uid()` in `portalData.ts`) for new rows; server may keep or replace, but must return the canonical id in the response. |
| Versioning | URL path (`/v1`), breaking changes → `/v2` |

## 3. Authentication & Authorization

The frontend has **no auth today** — add it as part of this build. Two audiences:

| Role | Capability | Endpoints |
|---|---|---|
| **Viewer** (any Premera/UST stakeholder) | Read report data, submit feedback, vote on rows | `GET /portal`, `POST /feedback`, `PUT /quick-feedback/{itemId}` |
| **Editor / Admin** (delivery leads, i.e. whoever currently has the `/admin` route) | Full read + write of report content | all endpoints |

Recommended: OAuth2/OIDC (Azure AD fits Premera) issuing JWT bearer tokens, role read from a
claim:

```
Authorization: Bearer <jwt>
```

- `GET` endpoints: require `viewer` or `admin`.
- `POST` / `PUT` / `PATCH` / `DELETE` on `/portal*`: require `admin`.
- `POST /feedback`, `PUT /quick-feedback/*`: require `viewer` or `admin`.
- All write actions go to an **audit log** (who, when, what changed) — see §8.

Return `401` when the token is missing/invalid, `403` when the role is insufficient.

## 4. Data Model — `PortalData` (primary dataset)

This is the **exact current TypeScript shape** the frontend consumes
(`src/data/portalData.ts`). Mirror it server-side field-for-field — this is what
`GET /portal` returns and `PUT /portal` accepts.

```ts
type StatusKey = 'onTrack' | 'atRisk' | 'complete' | 'delayed' | 'notStarted'

interface MilestoneRow {
  id: string
  label: string
  due: string        // free text, may be empty
  revised: string     // free text, may be empty
  status: StatusKey
}

interface MilestoneSection {
  id: string
  title: string        // e.g. "Critical Transition Milestones"
  rows: MilestoneRow[]
}

type ActionStatus = 'open' | 'inProgress' | 'closed'

interface RiskAction {
  id: string
  action: string        // mitigation action text
  owner: string
  dueDate: string        // free text
  actionStatus: ActionStatus
}

interface IssueRow {
  id: string
  issue: string                  // "Issue / Risk / Dependency" text
  mitigations: string[]           // "Mitigation Plan / Next Steps" bullets
  date: string                    // free text, e.g. "2/24"
  status: StatusKey
  needsPremeraAttention?: boolean  // flags row in Risks & Actions page + modal banner
  riskActions?: RiskAction[]       // shown in RiskActionsModal "Actions & Ownership" table
}

interface TowerGoLive {
  id: string
  name: string         // "ITO" | "BPO" | ...
  goLive: string        // free text date
  status: StatusKey
}

type WorkstreamIcon =
  | 'share2' | 'userCog' | 'arrowLeftRight'
  | 'truck' | 'building2' | 'shieldCheck' | 'clipboardCheck'

interface Workstream {
  id: string
  icon: WorkstreamIcon
  name: string
  status: StatusKey
  accomplishments: string[]
  activities: string[]
}

interface ProgramTab {
  levelLabel: string         // e.g. "<<ITO Program Level>>"
  workstreams: Workstream[]
  towers: TowerGoLive[]
  milestones: MilestoneSection[]
  issues: IssueRow[]
}

interface PortalData {
  asOf: string                // slide header "AS OF" pill, free text e.g. "DD/MM/YYYY"
  program: {
    ito: ProgramTab
    bpo: ProgramTab
  }
  tower: {
    levelLabel: string         // e.g. "<<Tower Level>>"
    accomplishments: string[]
    activities: string[]
    milestones: MilestoneSection[]
    keyActivities: MilestoneSection[]
    issues: IssueRow[]
  }
}
```

### Status enum reference (`src/data/status.ts`)

| `StatusKey` | Label | Dot colour |
|---|---|---|
| `onTrack` | On Track | `#1aa260` (green) |
| `atRisk` | At Risk | `#f2b705` (amber) |
| `complete` | Complete | `#8e44ad` (purple) |
| `delayed` | Delayed | `#e0392b` (red) |
| `notStarted` | Not Started | `#b0b6bd` (grey) |

Server must **reject** any `status` outside this set with `422`.

### Page → data mapping (so the API covers every screen)

| Route | Page | Reads from `PortalData` |
|---|---|---|
| `/program` | `ProgramLevel.tsx` | `program.ito` and `program.bpo` (tab-switchable): `levelLabel`, `workstreams`, `towers`, `milestones` |
| `/tower` | `TowerLevel.tsx` | `tower`: `levelLabel`, `accomplishments`, `activities`, `milestones`, `keyActivities` |
| `/risks` | `Risks.tsx` | `program.ito.issues`, `program.bpo.issues`, `tower.issues` — filtered to `atRisk`/`delayed`, each row opens `RiskActionsModal` showing `mitigations` + `riskActions` |
| `/feedback` | `Feedback.tsx` | none (separate `FeedbackEntry` resource, §6) |
| `/admin` | `Admin.tsx` | full `PortalData` (read + write), plus JSON export/import of the same shape |

## 5. Endpoints — `/portal` (primary dataset)

### 5.1 Whole-document (minimum viable backend)

#### `GET /v1/portal`
Returns the full `PortalData` snapshot. Called on load by every slide page via
`usePortalData()`.

```http
GET /v1/portal
Authorization: Bearer <jwt>
```
**200 OK**
```json
{
  "asOf": "06/12/2026",
  "program": {
    "ito": { "levelLabel": "...", "workstreams": [...], "towers": [...], "milestones": [...], "issues": [...] },
    "bpo": { "levelLabel": "...", "workstreams": [...], "towers": [...], "milestones": [...], "issues": [...] }
  },
  "tower": { "levelLabel": "...", "accomplishments": [...], "activities": [...], "milestones": [...], "keyActivities": [...], "issues": [...] }
}
```

#### `PUT /v1/portal`
Replaces the full document — exactly what the Admin **Save** button sends today
(`save(draft)` in `PortalDataContext`).

```http
PUT /v1/portal
Authorization: Bearer <jwt>
Content-Type: application/json

{ "asOf": "...", "program": { "ito": { ... }, "bpo": { ... } }, "tower": { ... } }
```
**200 OK** → returns the saved document (with server-canonicalised ids).
**422** → validation error (§7).
**403** → caller is not `admin`.

> Concurrency: support optimistic locking via `ETag`/`If-Match` or a top-level `version`
> integer — two admins could otherwise clobber each other's edits, since the Admin page
> edits a full local draft (`structuredClone`) and overwrites on Save with no merge.

#### `POST /v1/portal/reset`
Restores the seeded defaults — backend equivalent of the Admin **Reset** button
(`reset()` → `DEFAULT_PORTAL_DATA`).

**200 OK** → returns the restored document. `admin` only.

| `GET/POST` | `/portal/program/{programKey}/workstreams` | List / create workstreams |
| `PUT/DELETE` | `/portal/program/{programKey}/workstreams/{id}` | Update / delete |
| `GET/POST` | `/portal/program/{programKey}/towers` | List / create tower go-live rows |
| `PUT/DELETE` | `/portal/program/{programKey}/towers/{id}` | Update / delete |
| `GET/POST` | `/portal/program/{programKey}/milestones` | List / create milestone sections |
| `PUT/DELETE` | `/portal/program/{programKey}/milestones/{sectionId}` | Update / delete section (with its rows) |
| `GET/POST` | `/portal/program/{programKey}/issues` | List / create issues |
| `PUT/DELETE` | `/portal/program/{programKey}/issues/{id}` | Update / delete |
| `GET/POST` | `/portal/program/{programKey}/issues/{issueId}/actions` | List / create risk actions on an issue |
| `PUT/DELETE` | `/portal/program/{programKey}/issues/{issueId}/actions/{actionId}` | Update / delete a risk action |

```http
POST /v1/portal/program/ito/workstreams

{
  "icon": "shieldCheck",

}
```
**201 Created**
```json
{ "id": "b2f1...", "icon": "shieldCheck", "name": "Information Security", "status": "onTrack", "accomplishments": ["..."], "activities": ["..."] }
```

#### Example — add a risk action to an issue (drives `RiskActionsModal`)
```http
POST /v1/portal/program/ito/issues/pi1/actions
Content-Type: application/json
```

```http
PUT /v1/portal/program/ito/issues/pi3
Content-Type: application/json

{ "issue": "...", "mitigations": ["..."], "date": "3/10", "status": "atRisk", "needsPremeraAttention": true }
```

**200 OK** → updated `IssueRow`.

## 6. Endpoints — `/feedback` (Feedback to UST page)

`Feedback.tsx` currently keeps submissions in React state only (`useState<Entry[]>([])`) —
**they vanish on every page reload.** Backing this with a real resource is one of the most
impactful gaps to close.

```ts
interface FeedbackEntry {
  id: string
  rating: number            // 1–5, from the star picker
  area: string               // one of the AREAS list (free text accepted server-side, see note)
  priority: 'Low' | 'Medium' | 'High' | 'Critical'
  subject: string             // required, non-empty
  details: string             // optional
  submittedBy?: string         // from auth token if available; anonymous otherwise
  createdAt: string            // ISO-8601, server-assigned
}
```

`area` is chosen from a fixed dropdown in the UI today (`AREAS` constant in
`Feedback.tsx`): `Application Support`, `Application Development`, `Claims Operations`,
`Provider Data Management`, `Database Administration`, `End-to-End Testing`,
`Enrollment & Billing`, `Product Configuration`, `Microsoft 365 Support`,
`Provider Appeals & Grievances (Appian)`, `Provider Contact Center`,
`Service Desk / IT Operations Center`. The API should validate against this list (or a
server-managed equivalent) but doesn't need to hardcode it permanently — expose it via
`GET /feedback/areas` so the list can be changed without a frontend deploy.

| Method | Path | Description |
|---|---|---|
| `POST` | `/feedback` | Submit feedback → `201` + created entry |
| `GET` | `/feedback` | List entries, newest first. Supports `?area=`, `?priority=`, `?limit=`, `?cursor=` |
| `GET` | `/feedback/summary` | `{ "count": 12, "average": 4.4 }` — powers the log panel's average-rating header |
| `GET` | `/feedback/areas` | `string[]` — current valid `area` values for the dropdown |

#### Example
```http
POST /v1/feedback
Content-Type: application/json

{ "rating": 5, "area": "Application Support", "priority": "Medium",
  "subject": "Faster P2 turnaround", "details": "EDI gateway incident handled well." }
```
**201 Created**
```json
{ "id": "fb_91a...", "rating": 5, "area": "Application Support", "priority": "Medium",
  "subject": "Faster P2 turnaround", "details": "EDI gateway incident handled well.",
  "createdAt": "2026-06-16T14:32:00Z" }
```

## 7. Endpoints — `/quick-feedback` (per-row thumbs up/down widget)

The `QuickFeedback` component (used inline on every `MilestoneTable` and `IssuesTable` row,
in both Program and Tower views) lets a viewer thumbs-up/down a specific row and optionally
leave a one-line comment on a thumbs-down. Today this is **per-browser only**
(`localStorage` key `ustpremera_quickfeedback_v1`, keyed by `itemId`) — delivery leads on a
different machine never see these votes. This needs to become shared.

```ts
interface QuickFeedbackEntry {
  itemId: string            // the row's id, e.g. a MilestoneRow.id or IssueRow.id
  vote: 'up' | 'down'
  comment?: string            // only meaningful on a 'down' vote
  submittedBy?: string         // from auth token if available
  updatedAt: string            // ISO-8601
}
```

| Method | Path | Description |
|---|---|---|
| `GET` | `/quick-feedback` | List all entries (e.g. for an Admin "concerns" dashboard). Supports `?vote=down` |
| `GET` | `/quick-feedback/{itemId}` | Get the current viewer's vote on a row (matches `readEntry()` behavior) |
| `PUT` | `/quick-feedback/{itemId}` | Upsert `{ vote, comment? }` — matches `writeEntry()` |
| `DELETE` | `/quick-feedback/{itemId}` | Remove the vote — matches `deleteEntry()` (clicking the active thumb again) |

> Decide whether a vote is **per-user** (one vote per `itemId` per identity, requiring the
> `viewer` role) or an open tally (multiple viewers vote, aggregated). The current
> single-browser model behaves like "one vote per browser" — per-user with auth is the
> closest real equivalent and prevents accidental overwrites between stakeholders.

#### Example
```http
PUT /v1/quick-feedback/pmr8
Content-Type: application/json

{ "vote": "down", "comment": "Contractor Transition date looks optimistic given current staffing." }
```
**200 OK**
```json
{ "itemId": "pmr8", "vote": "down", "comment": "Contractor Transition date looks optimistic given current staffing.", "updatedAt": "2026-06-16T14:40:00Z" }
```

## 8. Validation

Reject with `422 Unprocessable Entity` using the error envelope (§9) when:

- `status` ∉ the five `StatusKey` values.
- `icon` ∉ the seven `WorkstreamIcon` values.
- `actionStatus` ∉ `open | inProgress | closed`.
- `rating` ∉ `1..5`.
- `priority` ∉ `Low | Medium | High | Critical`.
- `vote` ∉ `up | down`.
- Required string fields (`name`, `issue`, `subject`, `action`) are empty after trim.
- A `PUT /portal` body is missing `program.ito`, `program.bpo`, or `tower`.
- `programKey` path param ∉ `ito | bpo`.

Free-text date fields (`due`, `revised`, `date`, `dueDate`, `goLive`, `asOf`) are
intentionally **not** format-validated — they render verbatim on the slides exactly as
typed in Admin.

---

## 9. Error format

All errors use one envelope:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "status must be one of onTrack, atRisk, complete, delayed, notStarted",
    "field": "program.ito.workstreams[2].status",
    "requestId": "req_8f2c..."
  }
}
```

| HTTP | `code` | When |
|---|---|---|
| 400 | `BAD_REQUEST` | Malformed JSON |
| 401 | `UNAUTHENTICATED` | Missing/invalid token |
| 403 | `FORBIDDEN` | Authenticated but not `admin` (on write endpoints) |
| 404 | `NOT_FOUND` | Unknown id |
| 409 | `CONFLICT` | Optimistic-lock/version mismatch on `PUT /portal` |
| 422 | `VALIDATION_ERROR` | Failed a rule in §8 |
| 500 | `INTERNAL` | Unexpected server error |

## 10. Audit log

Every write (`PUT /portal`, any `/portal/**` mutation, `PUT/DELETE /quick-feedback/*`)
should append an immutable record:

```ts
interface AuditEntry {
  id: string
  actor: string            // user id / email from JWT, or "anonymous"
  action: 'create' | 'update' | 'delete' | 'reset'
  resource: string          // e.g. "portal.program.ito.issues.pi3"
  diff?: object              // before/after, or a JSON patch
  timestamp: string          // ISO-8601
}
```

| Method | Path | Description |
|---|---|---|
| `GET` | `/audit` | List audit entries, newest first. Supports `?resource=`, `?actor=`, `?since=` |

## 11. Frontend integration — what changes, what doesn't

No slide component changes are required. Only the data layer changes:

1. **`src/context/PortalDataContext.tsx`**
   - Replace `load()` (currently reads `localStorage`) with `await fetch(`${BASE}/portal`)`
     on mount; show a loading state while pending.
   - In `save(next)`, call `await fetch(`${BASE}/portal`, { method: 'PUT', body: JSON.stringify(next) })`.
   - In `reset()`, call `POST /portal/reset` instead of restoring the in-memory
     `DEFAULT_PORTAL_DATA` constant.
   - Optionally keep `localStorage` as an offline cache / optimistic fallback.
2. **`src/components/QuickFeedback.tsx`**
   - Replace `readEntry`/`writeEntry`/`deleteEntry` (currently `localStorage`) with
     `GET/PUT/DELETE /quick-feedback/{itemId}`.
3. **`src/pages/Feedback.tsx`**
   - On submit, `POST /feedback` instead of pushing to local `log` state.
   - On mount, `GET /feedback` to populate the log panel (currently empty until the user
     submits something in that session) and `GET /feedback/summary` for the average-rating
     header.
   - Populate the `AREAS` dropdown from `GET /feedback/areas` instead of the hardcoded
     constant (or keep the constant as a fallback/default seed).
4. **Auth** — add a token provider (e.g. MSAL for Azure AD) and attach the bearer header in
   a small `apiFetch` wrapper used by all of the above.
5. **Config** — add `VITE_API_BASE_URL` to `.env`, read via
   `import.meta.env.VITE_API_BASE_URL`.

The Admin page's existing **Export/Import** JSON buttons need no change — they already
produce/consume the exact `PortalData` shape and remain useful for seeding lower
environments via `PUT /portal`.

## 12. Suggested implementation

- **Stack:** contract is language-agnostic. Natural fits: .NET (Premera-aligned),
  Node/Express, or FastAPI.
- **Persistence:**
  - `PortalData` — a single JSON/document store (Cosmos DB, MongoDB, or Postgres `jsonb`
    column) maps 1:1 to the shape in §4.
  - `FeedbackEntry`, `QuickFeedbackEntry`, `AuditEntry` — relational tables (each is a flat,
    queryable record with filters/pagination).
- **Caching:** `GET /portal` is read-heavy and small — cache with a short TTL, bust on
  write.
- **OpenAPI:** generate a Swagger/OpenAPI 3 spec from these endpoints for client codegen
  and UAT reviewers.
